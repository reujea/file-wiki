# 병렬화 벤치마크 캐시 편향

## 상황
100문서 병렬 벤치마크에서 workers=1/2/4를 순차 실행. 1차 결과 workers=4가 1.31x 빠름 → "병렬화 성공" 보고.

## 문제
3회 중앙값 측정 시 workers=1(5.83s) ≥ workers=4(6.04s). 이전 결과는 거짓.

## 원인
1회차에서 workers=1은 디스크 캐시 cold, workers=4는 warm. **순차 실행 순서에 따른 캐시 편향**이 1.31x를 만들어냄. 또한 stub LLM의 per-doc 가공 시간(60ms)이 Mutex 경합 비용보다 짧아 병렬화 이득 없음.

## 개선
- 성능 벤치는 **최소 3회 실행 중앙값** 사용. 1회 결과는 참고용으로만 취급
- 병렬화는 **가공 시간 > 직렬화 지점 비용**일 때만 유효 (실 LLM 25초에서는 효과 예상)
- 벤치마크 구조: 각 조건을 독립 실행하거나, warm-up 라운드 포함

## 교훈
단일 벤치 실행으로 성능 결론을 내리지 않는다. 캐시(디스크/CPU/OS), 실행 순서, 시스템 부하 등 외부 요인이 결과를 왜곡할 수 있다.
