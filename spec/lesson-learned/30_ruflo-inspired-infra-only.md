# Lesson 30 — Ruflo 영감 개선: 인프라만 선구현 + 디폴트 비활성 (lesson 15 패턴 재적용)

## 상황 (2026-05-15)

ruvnet/ruflo 분석 후 6건 개선 후보를 도출. 즉시 적용 가능 3건 (A1 LLM 캐시, A2 KG hop, B1 다양성) 구현 진행.

3건 모두 "효과는 5K 코퍼스 측정 후 결정" 상태인데, 인프라 추가는 빠를 수 있고 후속 트리거 시 코드 수정 없이 켤 수 있도록 마무리해야 했다.

## 문제

A1/A2/B1을 "기능까지 모두 켜진 상태"로 도입하면:
- 5K 코퍼스 없이 디폴트로 켜져 측정 편향 (lesson 4 캐시 편향과 유사 위험)
- 미검증 동작이 사용자 검색 결과에 즉시 영향 → 회귀 책임 분산
- 트리거 도달 후 평가 시 "이미 켜져 있어서 비교 baseline 부재"

반대로 "효과 측정 후 코드 작성"으로 미루면:
- 트리거 도달 시점에 코드 충돌·재구현 비용
- lesson 14 (미연결 포트) 재발 — 인프라만 둔 채 호출 누락

## 원인 (메타 룰)

기능 도입은 두 단계로 분리해야 한다:

1. **인프라 단계** — config 필드 + 데이터 구조 + 통합 코드 작성. 디폴트 = 비활성 (값 0).
2. **활성화 단계** — 5K 코퍼스 측정 + 디폴트 변경 (별도 트리거).

lesson 15 (MinHash 포트 호출처 0건) 의 재발 방지 형태를 그대로 적용.

## 개선

### 적용 패턴

| 항목 | 인프라 (이번 작업) | 활성화 (트리거 대기) |
|------|---------------------|----------------------|
| A1 LLM 캐시 | settings.db 테이블 + 3 API + 단위 테스트 3 | claude_cli 호출 직전 lookup/upsert 통합 |
| A2 KG hop | SearchConfig.expand_kg_hops + McpState 필드 + handle_search 후처리 | 디폴트 0 → 1~2로 변경 |
| B1 다양성 | SearchConfig.diversity_threshold + McpState 필드 + swap 로직 | 디폴트 0 → 3으로 변경 |

핵심: **두 곳에 McpState 인스턴스 생성처가 있음** (cli/main.rs:914, shared/cli.rs:363). 양쪽 모두 cfg.search.* 주입 (lesson 26 schema 이중 정의 패턴 재발 방지).

### 체크리스트 (신규 인프라 도입 시)

1. config 필드 추가 → 디폴트 = "비활성"으로 의미하는 값 (0, false, empty)
2. McpState 등 핵심 struct에 필드 추가
3. **모든 인스턴스 생성처에 새 필드 주입** (grep으로 확인)
4. 통합 코드 (후처리, hook, 분기) 작성 — 비활성 시 비용 0
5. 단위 테스트로 "디폴트 = 비활성" 단언
6. 활성화 트리거 명시 (예: "5K 코퍼스 측정 후 디폴트 변경 검토")
