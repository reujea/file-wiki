## 상황 (Phase 66, 2026-05-04)

Phase 65에서 7탭 → 3계층 IA(입력·가공/검색·MCP/운영) 17서브탭으로 재설계. 빌드·배포 후 사용자 피드백으로 7탭 IA로 원복하면서 동시에 Pipeline 탭 내부를 새 구조(좌 사이드바 + 가운데 3탭 + 우 인스펙터)로 재설계 결정. 기존 Phase 65 코드를 일부 원복하고 일부 신규 구조에 흡수해야 하는 복합 작업.

## 문제

대규모 IA 변경의 원복은 단순 git revert가 아니다. Phase 65에서 만든 부분(fastembed 고정 / dead config 정리 / lesson 20)은 유지 가치가 있고, IA 구조(NAV_GROUPS / switchGroup / 13 신규 서브탭 컨테이너)만 원복해야 한다. 동시에 새 구조(Pipeline 가운데 3탭 + 우 인스펙터 + 검색 파이프라인 17노드 + 배치 설정)를 추가하면서 Phase 65 신규 서브탭 라우팅이 죽지 않게 하기 위한 정합성 작업이 동반된다.

## 원인

1. **부분 원복의 경계 모호성**: 어느 부분을 살리고 어느 부분을 되돌릴지의 결정이 사용자 검증 후에야 명확해짐. 첫 IA 결정 단계에서 사용자에게 와이어프레임 수준 미리보기를 제공하지 않은 것이 원인.
2. **Phase 65 작업의 결합도**: NAV_GROUPS 라우팅 / Settings 탭 폐기 / 신규 서브탭 13개 / fastembed 고정 / dead config 정리가 모두 한 phase에 묶여서 부분 원복 시 의존성을 따라가야 함.
3. **사용자 피드백 시점**: 빌드 배포 후 화면 직접 확인 시점에야 "원복하자" 결정. 이전 wireframe 단계에서 결정 가능했을 사항.

## 개선

### 패턴 A — 사용자 피드백 사이클을 IA 작업 전에 배치

```
1. ASCII 와이어프레임 + 결정사항 명시 (Pipeline 탭 내부 구조까지)
2. 사용자 검토 → 합의
3. 빌드 후 화면 확인
4. (필요 시) 원복 또는 미세 조정
```

Phase 65에서는 1~2단계 후 바로 구현했는데, **3단계에서 사용자가 "이거 아니다" 결정 가능**한 IA 변경은 와이어프레임 수준 검토를 한 번 더 거치는 것이 안전.

### 패턴 B — 부분 원복 시 살릴 것/되돌릴 것 명시 분리

원복 결정 시점에 다음 4축을 명시:
1. UI 구조 (탭/nav) — 되돌림
2. 데이터 모델 변경 (config.rs default / dead config) — 유지
3. 신규 함수/렌더 함수 — 일부 흡수, 일부 폐기
4. 문서/lesson — 유지 (학습 가치)

Phase 66에서는 이 4축을 명시한 결정 매트릭스로 작업 시작. UI는 원복, 데이터 모델은 유지, 함수는 새 구조에 흡수, lesson 20 유지 + lesson 21 신규.

### 패턴 C — IA 변경 시 "구조" + "내용물" 분리

```
구조 (탭 nav, 라우팅):
  외형 변경. 비교적 가볍게 변경/원복 가능.

내용물 (각 탭의 form, 데이터):
  config 섹션과 결합. 변경 비용 큼.
```

Phase 65에서 신규 서브탭 13개의 내용은 거의 다 기존 form 재사용(`renderSubtabConfig` 헬퍼 + `_renderField`). 즉 내용물은 거의 변경 없이 구조만 바뀐 형태였다. 이런 경우 원복 비용은 작음.

반면 Phase 66의 가운데 3탭은 검색 파이프라인 17노드 + 인스펙터라는 **새 내용물**을 도입해야 함. 구조 + 내용물 동시 변경이라 비용이 더 큼.

### 패턴 D — Pipeline 인스펙터 패턴

신규 도입한 우측 인스펙터(`pb-inspector`) 패턴은 다른 영역에도 재사용 가능:
- 가공 파이프라인 노드 클릭 → 인스펙터에 노드 설명 + 설정
- 검색 파이프라인 노드 클릭 → 인스펙터에 검색 단계 설명 + 트리거 placeholder
- 배치 설정 섹션 클릭 → 인스펙터에 config form

```
selectedInspector: 'process:nodeId' | 'search:nodeId' | 'batch:section'
```

이 패턴으로 가운데 영역은 "흐름 시각화" 역할, 우측은 "단일 항목 편집" 역할로 분리. 사용자 시선 이동이 좌 → 가운데 → 우 한 방향으로 흐름.

### 패턴 E — IA 작업의 빌드 비용 인식

| Phase | 빌드 비용 |
|-------|----------|
| Phase 65 | 워크스페이스 4분 40초 + Tauri 40분 (전체) → 후속 4분 + 4분 (증분) |
| Phase 66 | 워크스페이스 6초 + Tauri 4분 (증분) |

Phase 65는 fastembed 첫 빌드라 초기 비용 컸지만 Phase 66은 IA 변경 위주라 증분 빌드로 빠름. **IA 작업은 코드 변경량 대비 빌드 비용이 작은 편**이라는 점이 부분 원복/재구성을 가능하게 한 요인.

## 적용 범위

- 향후 IA 결정 시 **와이어프레임 검토 → 합의 → 구현 → 화면 확인** 4단계 적용
- 부분 원복 시 4축(UI 구조 / 데이터 모델 / 함수 / 문서) 매트릭스 명시
- Pipeline 인스펙터 패턴은 다른 복잡 영역(예: 토픽 편집, KG 시각화) 재사용 검토

## 사후 점검

- 사용자가 새 Pipeline 3탭 구조에 적응하는지 (가공/검색/배치 멘탈 모델 일치 여부)
- 검색 파이프라인 17노드의 placeholder 영역(HyDE / Parent / 단계별 trace)이 후속 phase로 자연스럽게 이어지는지
- Settings 3그룹(크레덴셜/일반/마이그레이션)이 사용자 멘탈 모델에 맞는지
