# 교차참조 성능 최적화 분석

## 현황

### 성능 회귀

| 규모 | 기존 처리량 | 현재 처리량 | 하락률 | 교차참조 관계 |
|------|-----------|-----------|--------|-------------|
| 100문서 | 133 docs/s | 6.3 docs/s | -95% | 386 |
| 1,000문서 | 92 docs/s | 2.4 docs/s | -97% | 3,699 |

### 병목 분석

**현재 알고리즘**: `auto_link()` (cross_reference.rs:276)

```
문서 1건 처리 시:
  1. vector_db.search_similar(embedding, top_k=3)  — O(N) SqliteVec 선형 스캔
  2. for each candidate (3건):
     - vector_db.get_keywords(id)                  — 1건당 1 API 호출
     - keyword overlap 계산                        — O(M×N) (HashSet으로 O(M))
     - relation type 판단                          — O(1)
  3. vector_db.link(doc_id, candidate_id)           — 1건당 1 호출
```

**시간 복잡도**: O(N) per doc × N docs = **O(N²)** (SqliteVec 선형 스캔이 원인)

| 구간 | 100문서 기여 | 1000문서 기여 |
|------|------------|-------------|
| search_similar (SqliteVec) | ~60% | ~75% |
| get_keywords × 3 | ~25% | ~15% |
| link + 저장 | ~10% | ~8% |
| keyword overlap | ~5% | ~2% |

## 20개 솔루션 벤치마킹

### 카테고리 1: ANN (Approximate Nearest Neighbor)

| # | 솔루션 | 작동 원리 | 쿼리 복잡도 | 10K 문서 속도 | 로컬 적합성 |
|---|--------|-----------|-----------|-------------|------------|
| 1 | **HNSW** | 계층적 스몰월드 그래프, greedy search | O(log N) | <1ms | **최적** |
| 2 | **FAISS IVF-PQ** | 클러스터링 + 양자화 압축 | O(N/nprobe) | <1ms | 적합 (10K 이하는 Flat 권장) |
| 3 | **Annoy** | 랜덤 프로젝션 트리 포레스트 | O(log N × n_trees) | <1ms | 적합 (인덱스 불변) |
| 4 | **ScaNN** | 비등방 양자화 + 파티셔닝 | O(N/partition) | <1ms | 조건부 (설정 복잡) |

## 권장 구현 계획

### Phase 1: 즉시 적용 — 예상 효과 10~50배

```
[기존] search_similar(embedding, 3) → O(N) 선형 스캔
                                       ↓
[개선 Step 0] 메타데이터 블로킹: 같은 doc_type 또는 30일 이내 문서만 후보
              → 비교 대상 N → N/5~N/10 축소

[개선 Step 1] 키워드 역색인: 키워드 3개 이상 공유하는 문서만 후보
              → 비교 대상 추가 축소

[개선 Step 2] 유사도 캐시: (doc_id_a, doc_id_b) → relation 캐시
              → 이미 비교한 쌍은 재계산 스킵

[개선 Step 3] get_keywords 배치화: 3건 개별 호출 → 1건 배치 조회
              → API 호출 3→1
```

### Phase 2: Qdrant HNSW 활용 — 예상 효과 100~1000배

```
[기존] SqliteVec 선형 스캔 O(N)
                     ↓
[개선] Qdrant HNSW O(log N) + payload 필터 (날짜/태그)
       → search_similar가 <1ms
```

### Phase 3: Hybrid 파이프라인 — 최종 아키텍처

```
[Stage 0] Blocking: 메타데이터 필터 (doc_type, ±30일, 공유 엔티티)
    ↓ N → N/5~N/10
[Stage 1] 벡터 검색: HNSW top_k=10 (~1ms)
    ↓ 10 후보
[Stage 2] 키워드 검증: overlap_min 체크
    ↓ 3~5 후보
[Cache] 이미 비교한 쌍 스킵
```
